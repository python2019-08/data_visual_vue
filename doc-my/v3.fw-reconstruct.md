# 1. 页面切换优化
```txt
现在有3个页面了。如果想 切换到第一个页面时，MainMiddle显示 地图 和地图上的一个 楼的 精模；切换到第二、第三个页面时，MainMiddle显示    楼的 精模的 某个楼层。

这种如何做更节省内存、且切换更快。​
```
 
          
这是一个非常棒的问题，它触及到了构建复杂三维 Web 应用的核心：**状态管理**和**性能优化**。

您的目标非常明确：在共享的 `MainMiddle` 组件中，根据当前路由（页面）动态地展示不同的三维内容，同时要保证内存占用低、切换速度快。

直接在每个页面里加载和卸载模型是**绝对不可取**的，因为这会导致大量的内存抖动和缓慢的加载时间。您提出的“如何做更节省内存、且切换更快”正说明您已经意识到了这一点。

### 最佳策略：集中管理，事件驱动

最优雅、最高效的解决方案是建立一个“中央-地方”的通信机制：

1.  **中央（`MainMiddle.vue`）**：作为三维场景的**唯一管理者**。它负责在应用启动时，**一次性加载所有**可能需要用到的三维模型（整栋楼、各个楼层等），但默认将它们全部设为**不可见**。它就像一个拥有所有道具的舞台监督。

2.  **地方（`HomePage.vue`, `Page01.vue`, `Page02.vue`）**：这些页面组件不直接接触任何三维操作。它们只负责在自己被激活时，向“中央”发送一个**信号**，告诉中央：“现在轮到我登场了，请布置适合我的场景。”

3.  **通信（`SceneController`）**：我们需要一个轻量级的“事件总线”或“状态控制器”，让“地方”可以方便地向“中央”发送信号。

### 实施步骤

我们将通过三个步骤来实现这个优雅的架构：

#### 第 1 步：创建场景状态控制器

我们先创建一个简单的、全局响应式的状态对象，用于通信。

```typescript /mnt/disk2/abner/zdev/jobs/task/data_visual_vue/src/utils/sceneController.ts
import { reactive } from 'vue'

/**
 * 定义场景可能的状态
 * - 'HOME': 显示地图和完整的楼宇模型
 * - 'FLOOR_1': 显示一楼模型
 * - 'FLOOR_2': 显示二楼模型
 */
export type SceneState = 'HOME' | 'FLOOR_1' | 'FLOOR_2'

export const sceneState = reactive<{
  activeView: SceneState
}>({
  activeView: 'HOME', // 默认状态
})
```

#### 第 2 步：改造 `MainMiddle.vue` (中央舞台监督)

让 `MainMiddle.vue` 负责加载所有模型，并根据 `sceneState` 的变化来更新场景。

```vue
// src/components/MainMiddle.vue

import { onMounted, watch } from 'vue'
import * as Cesium from 'cesium'
import { sceneState, type SceneState } from '@/utils/sceneController' // 引入控制器

// ...

onMounted(() => {
  const viewer = new Cesium.Viewer(/*...*/)

  // --- 关键：在这里一次性加载所有模型，并设为不可见 ---
  const buildingModel = viewer.entities.add({
    name: '整栋楼',
    position: Cesium.Cartesian3.fromDegrees(/* ...经纬高... */),
    model: {
      uri: '/path/to/your/building.glb',
      show: false, // 默认不可见
    },
  })

  const floor1Model = viewer.entities.add({
    name: '一楼',
    position: Cesium.Cartesian3.fromDegrees(/* ...经纬高... */),
    model: {
      uri: '/path/to/your/floor1.glb',
      show: false, // 默认不可见
    },
  })

  const floor2Model = viewer.entities.add({
    name: '二楼',
    position: Cesium.Cartesian3.fromDegrees(/* ...经纬高... */),
    model: {
      uri: '/path/to/your/floor2.glb',
      show: false, // 默认不可见
    },
  })

  // --- 关键：监听场景状态变化 ---
  watch(() => sceneState.activeView, (newView: SceneState) => {
    // 1. 首先，隐藏所有内容
    viewer.scene.globe.show = false // 隐藏地球
    buildingModel.show = false
    floor1Model.show = false
    floor2Model.show = false

    // 2. 然后，根据新状态显示对应的内容
    switch (newView) {
      case 'HOME':
        viewer.scene.globe.show = true // 显示地球
        buildingModel.show = true
        // 可选：飞行到适合观察整栋楼的视角
        viewer.flyTo(buildingModel)
        break
      case 'FLOOR_1':
        floor1Model.show = true
        // 可选：飞行到适合观察一楼的视角
        viewer.flyTo(floor1Model)
        break
      case 'FLOOR_2':
        floor2Model.show = true
        // 可选：飞行到适合观察二楼的视角
        viewer.flyTo(floor2Model)
        break
    }
  }, { immediate: true }) // immediate: true 确保初始状态被正确设置

  // ... 其他相机限制代码 ...
})
```
#### 第 3 步：AppMenu.vue 的新职责：发号施令者 (The Commander)

1. AppMenu.vue 的新职责：发号施令者 (The Commander)
我们重构之后，AppMenu.vue 的角色变成了一个纯粹的“发号施令者”。
当用户点击菜单时，它只负责做一件事：更新 sceneState.activeView 的值。
它就像一个将军，只下达“目标：一楼！” (sceneState.activeView = 'FLOOR_1') 的命令。它不关心也不知道这个命令具体是如何被执行的。

2. MainMiddle.vue 的职责：场景执行者 (The Executor)
MainMiddle.vue 是三维场景的唯一管理者，是命令的“执行者”。
它内部的 watch 代码块，就好像是它的“耳朵”，在持续不断地监听 sceneState.activeView 这个状态。
一旦 watch 监听到 sceneState.activeView 的值发生了变化（比如从 'HOME' 变成了 'FLOOR_1'），它就会立即触发回调函数，执行具体的工作：隐藏所有模型，然后只显示一楼的模型，并把镜头飞过去。

**总结** 
我们可以把这个流程想象成这样：

用户点击“功能页面一”
AppMenu.vue (发令者)：立刻下达命令，将 sceneState.activeView 的值更新为 'FLOOR_1'。
MainMiddle.vue (执行者)：内部的 watch 立即“听到”了这个变化，触发执行，将三维场景切换到一楼的样子。 

而 MainMiddle.vue 中的 watch 代码，是**“接收和执行命令”的核心逻辑**，它不是冗余代码，而是我们整个响应式系统的关键所在。如果删除了它，命令虽然下达了，但没有人去执行，场景也就不会有任何变化了。

### 为什么这个方案能满足您的要求？

*   **节省内存**：所有模型只在应用启动时加载**一次**，并常驻内存。我们没有在页面切换时重复加载或卸载任何三维资产。
*   **切换更快**：页面切换时，我们做的只是改变几个 `boolean` 属性 (`model.show`)。这是一个极其轻量级的操作，几乎是瞬时完成的，因此切换速度非常快。
*   **代码解耦**：页面组件（`HomePage`等）完全不知道三维场景的具体实现细节，它们只负责“发号施令”。`MainMiddle` 则专注于三维渲染。这使得代码非常清晰，易于维护。

这个方案是目前处理此类需求的最佳实践。如果您准备好实施，我们可以从创建 `src/utils/sceneController.ts` 文件开始。
